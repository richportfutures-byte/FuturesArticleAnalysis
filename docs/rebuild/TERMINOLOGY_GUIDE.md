# Terminology Guide

## Preferred framing

Use language that reflects the current product honestly:
- market intelligence
- expert read
- bias brief
- contract implications
- confirmation watchlist
- invalidation watchlist
- alternative interpretation
- source grounding
- doctrine evaluation
- bounded use
- fail-closed behavior

## Use carefully

These terms are acceptable only with explicit limits:
- deployment
  - use it to mean bounded output handoff, not order routing or execution tooling
- pre-trade review
  - use it as inherited doctrine vocabulary, not as the primary product identity
- post-reaction review
  - use it as a retrospective workflow term, not as an automation claim
- simulated mode
  - use it only for explicit local or test-mode reasoning

## Avoid or deprecate

Avoid these as the default product frame:
- operator
- audit workflow as the primary product identity
- execution engine
- automated trading
- autonomous execution
- exact entry
- exact stop
- exact size
- chatbot-first workflow
- rules engine as the intelligence core

## Plain-language standard

When describing the app, the safest concise frame is:

This repo is a market-intelligence workbench that uses LLM reasoning for analysis and deterministic code only for guardrails, provenance, doctrine evaluation, bounded outputs, and fail-closed behavior.
